# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`EntsoE.jl` is a Julia client for the [ENTSO-E Transparency Platform RESTful API](https://transparencyplatform.zendesk.com/hc/en-us/sections/12783116987028-Web-API) (electricity market, generation, load, transmission, outage, balancing, master data for Europe). The package was scaffolded with [`OpenAPITemplate.jl`](https://github.com/langestefan/OpenAPITemplate.jl) (a PkgTemplates plugin around `openapi-generator`) on top of an OpenAPI 3.1 spec we derived from ENTSO-E's official Postman collection.

## Layout ŌĆö what is generated vs hand-written

```
src/
  EntsoE.jl              # main module (template-rewritten + ONE manual line ŌĆö see below)
  api/                   # generated by openapi-generator-cli ŌĆö DO NOT EDIT
    EntsoEAPI.jl         #   inner module
    apis/                #   one file per ENTSO-E group (Market, Load, ...)
    modelincludes.jl
  client/                # hand-written template overlay (Client, Auth, retry, etc.)
  conveniences/          # OUR ENTSO-E-specific helpers (do not regenerate)
    period.jl            #   entsoe_period(DateTime|Date|ZonedDateTime) -> Int64
    eic.jl               #   EIC named tuple of common bidding-zone codes
    client.jl            #   EntsoEClient + entsoe_apis
    conveniences.jl      #   umbrella include
gen/
  regenerate.jl          # rerun codegen against spec/openapi.json
  openapi-config.json
spec/
  openapi.json           # canonical OpenAPI 3.1 spec consumed by codegen
entsoe-openapi.{json,yaml}            # post-processed spec at repo root (see below)
Transparency Platform Restful API.postman_collection.json   # original ENTSO-E source
scripts/postman_to_openapi.jl         # converter (one-file Julia, self-installing deps)
```

The generated `src/api/` directory carries `linguist-generated=true` in `.gitattributes`. Re-running `gen/regenerate.jl` nukes and re-creates `src/api/` only ŌĆö nothing else.

### The one manual edit to a templated file

`src/EntsoE.jl` is a templated file but `gen/regenerate.jl` does not touch it. We added a single line at the end:

```julia
include("conveniences/conveniences.jl")
```

Anything we want re-exported from the package goes through that include. If you ever re-run the *full* PkgTemplates scaffold (rare), you'll need to re-add this line.

## Postman ŌåÆ OpenAPI conversion

ENTSO-E publishes its API exclusively through a Postman collection ŌĆö there is no official OpenAPI spec. The codegen pipeline needs OpenAPI, so we built a custom converter. Two artifacts at the repo root carry this:

- `Transparency Platform Restful API.postman_collection.json` ŌĆö the original Postman v2.1 export from ENTSO-E (~7.5 MB, 77 leaf requests across 8 groups). Treat it as the source of truth for parameter semantics: every parameter description starts with `[M]` (mandatory) or `[O]` (optional) and lists the allowed enum values inline (e.g. `[M] A25 = Allocation results`).
- `entsoe-openapi.json` and `entsoe-openapi.yaml` ŌĆö the post-processed OpenAPI 3.1 spec produced by `scripts/postman_to_openapi.jl`. `gen/regenerate.jl` consumes the JSON form (it gets copied into `spec/openapi.json` for codegen).

### Why we didn't use a generic PostmanŌåÆOpenAPI tool

We tried `postman2openapi` (the standard CLI). Result: every one of ENTSO-E's 77 requests collapsed into a single path `/` with one giant `GET` operation, because OpenAPI keys operations by `path + method` and ENTSO-E uses the same URL for everything (only the query string differs). One method on the generated client ŌĆö useless. Per-endpoint examples and `[M]`/`[O]` markers were also lost in the merge. That output sat at the same file paths (~10 MB YAML / ~7 MB JSON, mostly noise) before we replaced it.

### What `scripts/postman_to_openapi.jl` does instead

A single self-contained Julia script that reads the Postman JSON directly and emits a clean OpenAPI 3.1 spec with one operation per logical endpoint:

1. **Walks the Postman folder tree** recursively. Each leaf request keeps the top-level folder name (`Market`, `Load`, ŌĆ”) as its group.
2. **Synthesizes one path per request** of the form `/{group-slug}/{request-slug}` ŌĆö purely so the OpenAPI key uniqueness constraint is satisfied. The synthetic paths are *not* the real URL; `info.description` in the resulting spec calls this out explicitly so anyone running the spec through another tool understands.
3. **Translates per-parameter Postman metadata into OpenAPI fields.** `[M]` ŌåÆ `required: true`. The marker is stripped from the human-readable description but the rest (including `A25 = Allocation results` enum hints) is preserved verbatim. Numeric examples become `integer`, the rest stay `string`.
4. **Lifts `securityToken` out of every per-operation parameter list** into a single `components.securitySchemes` entry of type `apiKey, in: query, name: securityToken`, applied at the root via `security`. Generated clients then handle auth uniformly per-request (this is what `EntsoEClient`'s `pre_request_hook` keys off).
5. **Adds an `application/xml` 200 response** to every operation. ENTSO-E never returns JSON, so no schema is generated ŌĆö the success body comes through as a Julia `String`.

The script is dependency-self-bootstrapping: on first run it activates `~/.julia/environments/entsoe-scripts` and `Pkg.add`s `JSON3`, `YAML`, `OrderedCollections` once, so the file stays self-contained with no `Project.toml` beside it. Output sizes drop from postman2openapi's 10 MB YAML to ~180 KB because there's no parameter-union noise.

### Re-running it

If ENTSO-E publishes an updated Postman collection, drop it in at the repo root with the same filename and run:

```bash
julia scripts/postman_to_openapi.jl                       # rewrites entsoe-openapi.{yaml,json}
julia --project gen/regenerate.jl --from-file entsoe-openapi.json   # rebuilds src/api/
```

Custom paths are supported: `julia scripts/postman_to_openapi.jl <input.json> <output-base>`.

## Key API conventions worth remembering

- **One real endpoint, many synthetic paths.** ENTSO-E's API is a single URL dispatched by query parameters. We split it into 77 distinct OpenAPI operations under synthetic paths like `/market/12-1-d-energy-prices` purely so the codegen produces one Julia function per logical query. The actual HTTP call still lands on `https://web-api.tp.entsoe.eu/api?...`. See the **Postman ŌåÆ OpenAPI conversion** section above for the full rationale and the splitter script.
- **Auth is a query parameter, not a header.** The `securityToken` is appended to the URL. The scaffold's built-in `APIKey` auth is header-only and does NOT work for ENTSO-E. Use `EntsoEClient(token)` from `src/conveniences/client.jl` ŌĆö it installs a `pre_request_hook(ctx)` that injects `ctx.query["securityToken"]` whenever an op declares the `"SecurityToken"` requirement.
- **Periods are `Int64` UTC `yyyyMMddHHmm`.** Use `entsoe_period(::DateTime)` / `entsoe_period(::ZonedDateTime)` rather than building these by hand.
- **Domains/areas are EIC codes** (`10YNL----------L` etc.). A curated subset lives in the `EIC` named tuple; for less common zones pass the raw 16-char string.
- **Responses are XML strings.** Every operation's success type is `String` because every ENTSO-E response is `application/xml`. The package does not parse XML ŌĆö that's left to the user (or a future addition). This is also why `test/test-models.jl` skips its model-construction asserts: there are no `APIModel` subtypes to construct.

## How a typical query reads

```julia
using EntsoE
using Dates: DateTime

client = EntsoEClient(ENV["ENTSOE_API_TOKEN"])
apis   = entsoe_apis(client)

start_p = entsoe_period(DateTime("2023-08-23T22:00"))
stop_p  = entsoe_period(DateTime("2023-08-24T22:00"))

xml, _ = market121_d_energy_prices(
    apis.market, "A44", EIC.NL, EIC.NL, start_p, stop_p,
)
```

The generated function signatures all take `(<group>Api, required_positionals...; optional_kwargs...)`. The Postman collection is the authoritative source for which parameters are mandatory and what enum values they accept (look up `[M]`/`[O]` markers and the `description` fields in the original collection if signatures are ambiguous).

## When in doubt

- Need a specific endpoint's parameter list and code values? Inspect the original Postman collection ŌĆö `Transparency Platform Restful API.postman_collection.json`. The OpenAPI spec is derived from it but the Postman descriptions carry richer enum hints (`A25 = Allocation results` etc.).
- Need to know what `gen/regenerate.jl` will overwrite? Only `src/api/` ŌĆö never `src/EntsoE.jl`, `src/client/`, `src/conveniences/`, `test/`, or any of the docs.
